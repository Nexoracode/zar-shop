# دستورالعمل پروژه زر گالری

این فایل مرجع اصلی تمام Agentهایی است که روی این مخزن کار می‌کنند. قبل از هر تغییر، این فایل باید کامل خوانده و رعایت شود.

## کلیت پروژه

این پروژه یک فروشگاه آنلاین طلا با رابط فارسی و راست‌به‌چپ است.

- فریم‌ورک اصلی: Next.js با App Router و TypeScript
- فرانت‌اند و APIهای بک‌اند هر دو داخل Next.js پیاده‌سازی می‌شوند.
- ORM: Prisma
- دیتابیس: MySQL
- ساختار پروژه باید ماژولار، قابل نگهداری و قابل توسعه باقی بماند.
- رابط فروشگاه و پنل‌ها باید واکنش‌گرا، فارسی و RTL باشند.

## بخش‌های اصلی محصول

### فروشگاه

- نمایش و جست‌وجوی محصولات طلا
- صفحه جزئیات محصول و گالری تصویر/ویدئو
- قیمت‌گذاری با نرخ لحظه‌ای طلا
- سبد خرید و ثبت سفارش
- پرداخت آنلاین
- صدور و نمایش فاکتور رسمی فروشگاه

### حساب کاربری

- احراز هویت کامل و امن
- ثبت‌نام، ورود، خروج و مدیریت نشست
- تکمیل و ویرایش پروفایل
- مدیریت نشانی‌ها و اطلاعات موردنیاز فاکتور
- مشاهده سفارش‌ها، پرداخت‌ها و فاکتورها

### پنل مدیریت

- ثبت، ویرایش، انتشار و بایگانی محصول
- مدیریت دسته‌بندی، موجودی، وزن، عیار، اجرت، سود و مالیات
- گالری مرکزی برای تصاویر و ویدئوهای محصولات
- مدیریت سفارش و وضعیت ارسال
- مدیریت پرداخت‌ها و فاکتورها
- مدیریت کاربران، نقش‌ها و وضعیت حساب

## قواعد معماری

- صفحات و Route Handlerها در `src/app` قرار می‌گیرند.
- قابلیت‌های دامنه در `src/modules` تفکیک می‌شوند؛ مانند `auth`، `products`، `gold`، `orders` و `payments`.
- اجزای رابط قابل استفاده مجدد در `src/components` قرار می‌گیرند.
- ابزارهای عمومی، اتصال دیتابیس و تنظیمات محیطی در `src/lib` قرار می‌گیرند.
- مدل‌ها، migrationها و seed دیتابیس در `prisma` نگهداری می‌شوند.
- منطق تجاری نباید داخل کامپوننت‌های نمایشی یا Route Handlerها پراکنده شود؛ منطق قابل استفاده مجدد باید در service/module مناسب قرار بگیرد.
- قرارداد سرویس‌های بیرونی مانند درگاه پرداخت، منبع نرخ طلا و فضای ذخیره‌سازی باید قابل تعویض باشد.
- هیچ secret، رمز، API key یا اطلاعات واقعی مشتری نباید وارد Git شود.

## قواعد مالی و طلا

- مبلغ، وزن، نرخ طلا، اجرت، سود و مالیات در دیتابیس با نوع دقیق `Decimal` نگهداری شوند؛ برای داده مالی از `float` استفاده نشود.
- نرخ طلا فقط در سمت سرور دریافت شود و دارای timeout، cache و fallback کنترل‌شده باشد.
- نرخ طلای استفاده‌شده هنگام ایجاد سفارش باید در همان سفارش Snapshot شود.
- اجزای قیمت هر قلم شامل ارزش طلای خام، اجرت، سود، مالیات و مبلغ نهایی باید هنگام ثبت سفارش ذخیره شوند.
- تغییر نرخ طلا یا مشخصات محصول نباید مبلغ سفارش یا فاکتور قبلی را تغییر دهد.
- مبلغ پرداخت باید در callback مجدداً با سفارش تطبیق و نتیجه پرداخت سمت سرور verify شود.
- callback پرداخت باید idempotent باشد و تکرار آن باعث کاهش مجدد موجودی یا صدور چند فاکتور نشود.
- موجودی محصول فقط پس از پرداخت تأییدشده و در یک عملیات امن کاهش پیدا کند.
- اطلاعات حقوقی فروشنده و خریدار در فاکتور Snapshot شوند.

## امنیت و دسترسی

- همه ورودی‌های API با schema معتبرسنجی شوند.
- رمز عبور فقط به‌صورت hash امن ذخیره شود.
- نشست‌ها با cookie امن، `HttpOnly` و `SameSite` مناسب مدیریت شوند.
- کنترل نقش فقط در رابط کاربری کافی نیست؛ هر API مدیریتی باید مجوز کاربر را سمت سرور بررسی کند.
- نقش‌های اصلی پروژه `CUSTOMER`، `OPERATOR` و `ADMIN` هستند.
- عملیات حساس مدیریتی باید در Audit Log ثبت شوند.
- پیام خطا نباید secret، stack trace یا جزئیات داخلی دیتابیس را افشا کند.
- آپلود فایل باید محدودیت نوع، اندازه و نام امن داشته باشد.

## شیوه انجام هر تسک

1. قبل از شروع، وضعیت Git و فایل‌های مرتبط بررسی شوند.
2. تغییرات موجود کاربر حفظ شوند و فایل نامرتبط بازنویسی نشود.
3. تغییر در کوچک‌ترین محدوده لازم انجام شود.
4. بعد از انجام تسک، حداقل یک بررسی متناسب با تغییر اجرا شود.
5. برای تغییرات کد، در صورت ارتباط، این بررسی‌ها اجرا شوند:
   - `npm run typecheck`
   - `npm run lint`
   - تست‌های مرتبط
   - `npm run build` برای تغییرات پرریسک، ساختاری یا قبل از انتشار
6. برای تغییرات Prisma، علاوه بر بررسی TypeScript، `npm run db:generate` اجرا شود و schema اعتبارسنجی شود.
7. نتیجه بررسی‌ها همراه با چک‌لیست نهایی به کاربر گزارش شود.
8. قبل از commit باید صریحاً از کاربر تأیید گرفته شود.
9. تا قبل از تأیید کاربر هیچ commit ساخته نشود.
10. پس از تأیید کاربر، فقط فایل‌های مرتبط با همان تسک stage و با پیام روشن commit شوند.
11. تغییرات نامرتبط یا قبلی کاربر بدون اجازه وارد commit نشوند.

- [ ] فایل‌های نامرتبط تغییر نکرده‌اند.
- [ ] ساختار و قواعد دامنه پروژه رعایت شده‌اند.
- [ ] کنترل دسترسی، امنیت و اعتبارسنجی ورودی بررسی شده‌اند.
- [ ] محاسبات مالی و Snapshotها در صورت ارتباط بررسی شده‌اند.
- [ ] حداقل یک بررسی فنی اجرا و نتیجه آن ثبت شده است.
- [ ] TypeScript و lint برای تغییرات کد بدون خطا هستند.
- [ ] محدودیت‌ها، موارد باقی‌مانده و تنظیمات لازم شفاف گزارش شده‌اند.
- [ ] تأیید کاربر برای commit دریافت شده است.
- [ ] بعد از تأیید، فقط تغییرات مرتبط commit شده‌اند.

## قواعد Commit

- Agent نباید خودکار یا قبل از تأیید کاربر commit بزند.
- گزارش قبل از commit باید شامل خلاصه تغییرات، بررسی‌های اجراشده و فایل‌های موردنظر برای commit باشد.
- پس از تأیید، پیام commit باید کوتاه، مشخص و ترجیحاً با الگوی Conventional Commits باشد؛ مانند:
  - `feat(products): add product media gallery`
  - `fix(payments): make callback idempotent`
  - `docs(project): add repository workflow rules`
- هر commit باید یک موضوع منسجم داشته باشد.
- فایل‌های محیطی، secretها، خروجی build و فایل‌های آپلودشده نباید commit شوند.

## تعریف پایان کار

تسک زمانی آماده تأیید است که پیاده‌سازی کامل شده، بررسی متناسب اجرا شده، نتیجه و چک‌لیست ارائه شده و مورد باز یا ریسک شناخته‌شده اعلام شده باشد. پایان پیاده‌سازی به معنی اجازه commit نیست؛ commit فقط بعد از تأیید صریح کاربر انجام می‌شود.
